# Current State Review

Reviewed: 2026-03-19

---

## A. Executive summary

The repository is a coherent, working data engineering lab. Seven exercises
are implemented in both Python and JavaScript with 303 passing tests, zero
lint errors, zero type errors, and a polished end-to-end demo that processes
all four e-commerce sample datasets through ingestion, validation, cleaning,
and analytical querying. The code is well-structured, the demo path is clear,
and the documentation is extensive.

The repository has moved past prototype stage. It functions as a credible
learning lab and could serve as portfolio work. The main weaknesses are:
five placeholder modules with no implementation, some documentation that
drifts from actual numbers, and a few areas where the aspirational framing
outpaces what's actually built.

---

## B. What is working well

**Core implementations are solid.** All seven exercises (CSV ingestion, API
ingestion, validation framework, incremental ETL, CDC snapshot diff,
orchestration runner, observability tracker) work correctly in both languages
with meaningful test coverage including edge cases, failure recovery, and
rerun safety.

**The demo pipeline is the strongest feature.** A visitor can clone the repo,
run one command, and see 60 rows processed through ingestion, validation,
cleaning, and curated output — with a JSON manifest, observability metadata,
6 data-quality warnings, and 3 rejected rows. The analytics layer then loads
the output into SQLite and produces 5 analytical queries. Both Python and JS
produce identical results.

**Test quality is good.** 159 Python tests and 144 JavaScript tests. Coverage
includes malformed input, corrupted checkpoints, empty datasets, rerun
idempotency, golden output verification, JSON shape validation, and CDC
determinism. Tests are behavioral, not implementation-detail tests.

**Tooling is clean.** Ruff (lint + format), mypy (strict mode), and ESLint all
pass with zero errors. CI runs lint and tests as separate jobs with a demo
smoke test.

**Sample data is well-designed.** The four CSV files (customers, products,
orders, order_items) have intentional quality issues (duplicates, null
values, orphan FKs, negative prices, inconsistent dates, casing problems)
that are documented and used across multiple exercises.

**SQL assets are useful.** 27 scripts across DDL, DML, analytics, and
warehouse ETL — all SQLite-compatible, documented, and aligned with the
sample data.

**Convention consistency.** Result objects use `snake_case` in both languages.
Checkpoint format, status strings, and CLI flag naming are standardized
across all modules.

## C. What is weak or inconsistent

**Five placeholder modules exist but are empty.** `cli/`, `storage/`,
`streaming/`, `warehouse/`, and `utils/` directories exist in both Python and
JavaScript with only docstring-only `__init__.py` / `index.js` files. The
README's repository structure diagram lists them prominently, which creates
an impression of breadth that isn't backed by implementation.

**Documentation count drift.** Several docs referenced hardcoded test counts
(110, 116) that became stale as tests were added. This was partially fixed in
this review pass by removing hardcoded counts where possible. The
`docs/repo-audit.md` is a point-in-time snapshot that no longer reflects the
current state (it references 7 test files and 110/116 tests).

**The "Dual Implementation" table in the README is aspirational.** It lists
pandas, Polars, Arquero, Great Expectations, Prefect, kafka-python, etc. —
none of which are used anywhere in the codebase. The implementations use
only standard library modules. This table may mislead visitors about what's
actually in the repo.

**Lock files are gitignored.** Both `poetry.lock` and `yarn.lock` are in
`.gitignore`. This means builds aren't reproducible — different machines may
resolve different dependency versions. Acceptable for a learning repo, but
worth noting.

**The `configs/` and `scripts/` directories are empty.** They contain only
`.gitkeep` files and are documented in the README structure diagram. They
serve no current purpose.

## D. What is missing

**No streaming exercises.** Listed as planned in the README roadmap and
exercise index. The `streaming/` module directories exist as placeholders
but contain nothing. The sample data includes `sensor_events.json` which was
designed for streaming exercises.

**No warehouse loading CLI.** Also listed as planned. The SQL assets exist but
there's no programmatic way to run them from Python/JS beyond the analytics
module (which loads silver-layer CSVs, not raw data through the SQL DDL/DML
pipeline).

**No benchmark exercise.** Listed as planned. No implementation.

**Manifests are not generated by exercises.** The platform conventions doc
defines a manifest format, and the demo writes one, but the individual
exercises (01-07) don't produce manifests. The `data/manifests/` directory
exists but is only populated by the demo.

**No `conftest.py` or shared test fixtures.** Each Python test file
independently defines its own helpers and sample data. A shared fixture file
could reduce duplication (e.g., the `write_csv`, `write_jsonl`, and
`_produce_silver` helpers appear in multiple files).

## E. Risk areas

**`docs/repo-audit.md` is frozen in time.** It describes the state of the
repo from an earlier pass and contains stale numbers. If someone reads it
expecting current data, they'll be misled. Consider adding a date header or
archiving it.

**The Node.js `node:sqlite` dependency is experimental.** The analytics
module uses `node:sqlite` which is marked experimental in Node 22. It works
today but could break in future Node releases. This is documented in the
sqlite-analytics-demo doc but worth watching.

**CI does not cache dependencies.** Both Python and JS workflows install
from scratch on every run. Not a correctness issue, but CI runs are slower
than necessary.

**The orchestrated-workflow customer_etl is not integrated with the demo.**
Two overlapping patterns exist: `customer_etl.py` (uses the orchestration
runner for customer data) and `demo.py` (processes all 4 tables without the
orchestration runner). They solve similar problems differently, which could
confuse someone studying the repo.

## G. Overall assessment

The repository is a **coherent lab** — past the scaffold and prototype stages.

It has a clear purpose, consistent implementation across two languages, a
working demo pipeline, meaningful tests, clean tooling, and documentation
that mostly matches reality. A visitor can clone it, run the demo, study the
exercises, and learn real data engineering patterns.

The main gap between "coherent lab" and "strong portfolio-ready project" is
the placeholder modules and the aspirational framing in some docs. The
implementations that exist are solid; the problem is that the repo promises
more breadth than it currently delivers (streaming, warehouse CLI, benchmark).

**Strongest parts:** demo pipeline, sample data design, test coverage for
edge cases and rerun safety, convention consistency.

**Weakest parts:** empty placeholder modules, documentation count drift,
aspirational framing that oversells scope.

**Recommendation:** Ship what exists as M1. Cut the aspirational framing to
match reality. Implement streaming and warehouse exercises as M2.
