# Platform Conventions

Shared conventions across the Python and JavaScript implementations. These keep
the repository coherent as exercises are added.

---

## Field naming in result objects

All pipeline result objects, checkpoint files, and serialized summaries use
**`snake_case`** field names in both Python and JavaScript.

```python
# Python â€” dataclass fields
@dataclass
class RunSummary:
    pipeline_name: str
    records_processed: int
    duration_seconds: float
```

```javascript
// JavaScript â€” plain object
return {
  pipeline_name: pipelineName,
  records_processed: transformed.length,
  duration_seconds: elapsed,
};
```

Internal JavaScript variables and function parameters remain `camelCase` as is
idiomatic. The `snake_case` convention applies only to the **shape of data
objects** â€” the things that get serialized, compared across languages, or
returned as structured results.

Why: result objects are data, not code. They appear in JSON files, test
assertions, and cross-language comparisons. A single casing convention
eliminates a class of bugs where Python tests and JavaScript tests diverge.

## Status strings

Three status domains exist. Each uses a fixed set of lowercase strings.

| Domain | Values | Used by |
|--------|--------|---------|
| Pipeline | `"success"`, `"failed"` | Orchestration runner, RunTracker |
| Step | `"success"`, `"failed"`, `"skipped"` | Orchestration step results |
| Validation | `"passed"`, `"warning"`, `"failed"` | Validation runner |

Do not invent new status strings. If a module needs to report a status, use one
of these three domains.

## Pipeline result shape

Every pipeline entry point (e.g., `run_pipeline`, `run_incremental_etl`,
`run_api_pipeline`) returns a structured result object. All results should
include at least:

| Field | Type | Description |
|-------|------|-------------|
| `pipeline_name` | string | Identifies which pipeline produced this result |
| `run_at` | string | ISO 8601 timestamp of when the run started |
| `duration_seconds` | float | Wall-clock time for the run |
| `status` | string | One of the pipeline status values |

Additional fields are module-specific (e.g., `records_seen`, `files_processed`,
`pages_fetched`). Use `records_` prefixed fields for row-level counts and
`files_` prefixed fields for file-level counts.

## Checkpoint format

Checkpoints are JSON files stored in `data/checkpoints/`. All checkpoints
follow this base structure:

```json
{
  "pipeline_name": "events_etl",
  "last_run_at": "2024-06-01T10:00:00Z",
  "total_runs": 3
}
```

Module-specific fields are added alongside these base fields. For example, the
incremental ETL checkpoint adds `processed_ids`.

Rules:
- Field names use `snake_case`.
- Timestamps are ISO 8601 with timezone.
- The file is overwritten atomically on each successful run.
- If the checkpoint is missing, the pipeline starts fresh.
- If the checkpoint is corrupted (invalid JSON), the pipeline starts fresh
  and logs a warning.

## Manifest format

Manifests are JSON files stored in `data/manifests/`. They record what a
pipeline run produced so that downstream consumers know what to expect.

```json
{
  "pipeline_name": "csv_ingestion",
  "run_id": "20240601_120000",
  "created_at": "2024-06-01T12:00:00Z",
  "source": "data/sample/customers.csv",
  "output": "data/bronze/merged.csv",
  "row_count": 42,
  "schema_hint": ["customer_id", "first_name", "last_name", "email"],
  "status": "success"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `pipeline_name` | string | Which pipeline created this output |
| `run_id` | string | Unique identifier for the run |
| `created_at` | string | ISO 8601 timestamp |
| `source` | string | Input path or URL |
| `output` | string | Output path |
| `row_count` | integer | Number of rows in the output |
| `schema_hint` | string[] | Column names or top-level keys in the output |
| `status` | string | `"success"` or `"failed"` |

Manifests are generated by all major pipeline exercises. A shared utility
(`manifest.py` / `manifest.js`) provides `write_manifest` and
`read_manifest` functions that enforce this format consistently.

Modules that generate manifests:
- CSV ingestion (`csv_ingestion`)
- API ingestion (`api_ingestion`)
- Incremental ETL (`events_etl`)
- Streaming processor (`sensor_stream`)
- Warehouse loader (`warehouse`)
- End-to-end demo (`ecommerce_demo`)

Manifest writes are best-effort â€” if the target directory is not writable
(e.g. in test environments), the pipeline logs a warning and continues.

## Validation result shape

All validation checks return a `CheckResult` (Python) or plain object (JS)
with these fields:

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Name of the check (e.g., `"check_no_nulls"`) |
| `passed` | boolean | Whether the check passed |
| `severity` | string | `"WARNING"` or `"CRITICAL"` |
| `message` | string | Human-readable result |
| `failing_rows` | int[] | Zero-indexed row numbers that failed |

The validation runner aggregates check results into a report:

| Field | Type | Description |
|-------|------|-------------|
| `dataset_name` | string | Name of the dataset being validated |
| `total_checks` | integer | Number of checks executed |
| `passed` | integer | Checks that passed |
| `failed` | integer | Checks that failed |
| `warnings` | integer | Failed checks with WARNING severity |
| `critical_failures` | integer | Failed checks with CRITICAL severity |
| `status` | string | `"passed"`, `"warning"`, or `"failed"` |
| `checks` | CheckResult[] | Individual check results |

## Logging

**Python:** Use the standard `logging` module with `__name__` as the logger
name.

```python
import logging
logger = logging.getLogger(__name__)
```

**JavaScript:** Use `console` methods. No external logging library is required.

| Level | Python | JavaScript | When to use |
|-------|--------|------------|-------------|
| Debug | `logger.debug()` | `console.debug()` | Detailed diagnostics |
| Info | `logger.info()` | `console.info()` | Normal operation milestones |
| Warning | `logger.warning()` | `console.warn()` | Recoverable problems |
| Error | `logger.error()` | `console.error()` | Failures that stop processing |

Log at **info** level when: a pipeline starts, a pipeline completes, an output
file is written, or a summary is produced. Log at **warning** level when: a
file is skipped, a record is malformed, or a check fails. Log at **error**
level when: a pipeline cannot complete its primary task.

## Output directory conventions

| Layer | Path | Contents |
|-------|------|----------|
| Raw | `data/raw/{pipeline_name}/{run_id}/` | Untouched source data |
| Bronze | `data/bronze/{pipeline_name}/{run_id}/` | Lightly processed |
| Silver | `data/silver/{pipeline_name}/` | Cleaned, deduplicated |
| Gold | `data/gold/{pipeline_name}/` | Business aggregates |
| Checkpoints | `data/checkpoints/{pipeline_name}.json` | Pipeline state |
| Manifests | `data/manifests/{pipeline_name}_{run_id}.json` | Run metadata |

Output files within a run directory use descriptive names: `raw.json`,
`processed.json`, `output.jsonl`, `inserts.csv`, etc.

## CLI argument conventions

**Python CLIs** use `argparse` with `--kebab-case` flags:

```
--input-dir, --output, --required-columns, --page-size, --max-pages
```

**JavaScript CLIs** use `node:util.parseArgs` with `--kebab-case` flags:

```
--input-dir, --output, --required-columns, --page-size, --max-pages
```

Both languages use the same flag names for the same concepts. Short flags
(`-i`, `-o`, `-h`) are optional and module-specific.

## Naming conventions summary

| Context | Convention | Example |
|---------|-----------|---------|
| Python variables and fields | `snake_case` | `records_processed` |
| JavaScript variables and parameters | `camelCase` | `recordsProcessed` |
| Result object fields (both languages) | `snake_case` | `records_processed` |
| JSON file keys | `snake_case` | `pipeline_name` |
| CLI flags | `--kebab-case` | `--input-dir` |
| File names (Python) | `snake_case.py` | `csv_pipeline.py` |
| File names (JavaScript) | `kebab-case.js` | `csv-pipeline.js` |
| Status strings | lowercase | `"success"`, `"failed"` |
| Timestamps | ISO 8601 | `"2024-06-01T10:00:00Z"` |
