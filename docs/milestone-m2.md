# Milestone 2

Completed: 2026-03-21

---

## Purpose

Milestone 2 expands the repository from a core exercise lab into a more
complete data platform demonstration. Where M1 established seven exercises
and one demo, M2 adds event processing, warehouse workflows, performance
benchmarking, shared infrastructure (manifests, config, test helpers), and
a second showcase pipeline. The repository now covers batch, event, and
analytical patterns with real cross-cutting platform capabilities.

Manifests are now generated by 6 pipeline entry points (best-effort):
CSV ingestion, API ingestion, incremental ETL, streaming processor,
warehouse loader, and both demo pipelines.

### Config-driven execution

Three CLIs accept `--config` with a JSON file: streaming, warehouse, and
benchmark. Example configs are in `configs/`.

### CI improvements

- Poetry virtualenv caching (`actions/cache`)
- Yarn cache caching (`actions/cache`)
- `pipx install poetry` (faster than pip)
- `sql/**` added to path triggers
- Sensor demo added as second smoke test

## Current strongest workflows

1. **E-commerce demo** — the clearest entry point. Four tables through
   ingestion, validation, cleaning, and analytical queries. Produces
   identical results in both languages.

2. **Sensor pipeline demo** — demonstrates the orchestration runner in a
   real workflow with dead-letter routing, hourly aggregation, and manifest
   generation. Good contrast with the e-commerce demo.

3. **Warehouse CLI** — the only workflow that exercises the full ELT path:
   raw files → staging → star-schema dims/facts → analytical queries.
   Uses the existing SQL assets programmatically.

4. **Streaming processor** — the most educational single module. Covers
   per-event validation, deduplication, event-time watermarks, lateness
   classification, and dead-letter routing.

---

## Current architectural limits

- **Placeholder modules weaken trust.** `storage/`, `utils/`, and `cli/`
  exist as empty directories. They appear in the project structure but
  contribute nothing. They should either be implemented or removed.

- **Two demo paths can confuse.** The e-commerce demo uses direct function
  calls while the sensor demo uses the orchestration runner. The
  relationship is documented but requires reading the docs to understand.

- **No unified CLI.** Each module has its own CLI entry point. There is no
  `data-platform-lab run streaming` or `data-platform-lab run demo`
  command. Users must know the module path.

- **Benchmark results are machine-dependent.** The benchmark exercise
  measures real execution time, which varies across machines. Results
  are documented with caveats but cannot be asserted in tests.

- **Manifests are best-effort.** Pipelines wrap manifest writes in
  try/except so tests that run in temp directories don't fail. This
  means manifest generation is not verified by most tests.

- **`node:sqlite` is experimental.** The JavaScript analytics and warehouse
  modules depend on a feature that could change in future Node releases.

---

## Recommended Milestone 3 targets

1. **Remove or implement placeholder modules.** `storage/`, `utils/`, and
   `cli/` should either become real or be deleted. The current state is
   the repo's most visible weakness.

2. **Log parsing exercise.** Use `data/sample/logs.log` to build a
   structured extraction pipeline. The sample data exists; the exercise
   is well-scoped and would add a new data format (semi-structured text).

3. **Unified CLI.** Create a single entry point (`data-platform-lab run
   <workflow>`) that dispatches to the existing module CLIs. This would
   make the repository more accessible without changing module internals.

4. **Streaming windowed aggregation.** Extend the streaming processor
   with tumbling or sliding time windows. The watermark infrastructure
   already exists; windows are the natural next step.

5. **DuckDB analytical layer.** Add an alternative to SQLite for
   analytical queries. DuckDB handles CSV/Parquet natively and would
   demonstrate a different analytical pattern.

6. **CI lock file strategy.** Either commit lock files (for
   reproducibility) or document the trade-off explicitly. The current
   silent gitignore is the worst option.

## How to explore the repository at M2

**Quick demo:** Run the e-commerce demo, then the sensor demo.
```bash
cd python
poetry install
poetry run python -m data_platform_lab.demo
poetry run python -m data_platform_lab.sensor_demo
```

**Study exercises:** Follow the [roadmap](roadmap.md) from Exercise 01
through Exercise 09.

**Run the warehouse:** Load raw data into a star schema.
```bash
poetry run python -m data_platform_lab.warehouse.cli
```

**Run benchmarks:** Compare ingestion strategies.
```bash
poetry run python -m data_platform_lab.benchmark.cli --num-files 50
```

**Use config files:** Run a workflow from a config.
```bash
poetry run python -m data_platform_lab.streaming.cli --config ../configs/streaming.json
```

**Read the docs:** Start with [milestone-m1.md](milestone-m1.md) for
history, [platform-conventions.md](platform-conventions.md) for patterns,
and [current-state-review.md](current-state-review.md) for an honest
assessment.
